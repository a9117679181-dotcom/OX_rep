# OX - Этап 09. Пояснительная записка

## Сервер, индексатор и служба выплат

Дата: 09.06.2026  
Версия: v1  
Проект: OX, спортивный экспресс  
Стартовый продукт: Суперэкспресс 15  
Девиз: Честно. Прозрачно. Мгновенно.

---

## 1. Что находится в этом этапе

Этот этап описывает серверную часть OX.

Если смарт-контракты - это место, где хранятся деньги и фиксируются правила, то серверная часть - это рабочий центр проекта. Она собирает данные, считает билеты, строит Merkle tree (меркл-дерево), следит за выплатами, отправляет транзакции за пользователя и показывает данные сайту и админ-панели.

Главная задача backend (серверной части):

```text
связать пользовательский интерфейс, спортивные API, risk engine, смарт-контракты и выплаты в одну управляемую систему.
```

---

## 2. Главное решение

Для MVP нужна модульная серверная архитектура:

```text
API server + database + indexer + jobs + settlement engine + Merkle builder + payout executor + monitoring
```

Расшифровка:

- `API server (сервер API)` - отдает данные сайту и админке.
- `database (база данных)` - хранит тиражи, билеты, результаты, логи, статусы выплат.
- `indexer (индексатор)` - читает события из блокчейна и синхронизирует их с базой.
- `jobs (фоновые задачи)` - выполняют регулярные процессы: опрос спортивных API, проверка результатов, мониторинг.
- `settlement engine (модуль расчета)` - считает победителей после финализации результатов.
- `Merkle builder (построитель Merkle tree)` - строит список выплат и доказательства.
- `payout executor (исполнитель выплат)` - отправляет `claimFor` и платит газ за пользователя.
- `monitoring (мониторинг)` - следит за ошибками, газом, API, очередями и аномалиями.

---

## 3. Почему серверная часть критична

Без backend OX не сможет:

- создать и подготовить тираж;
- подобрать 15 событий;
- рассчитать системные ставки;
- проверить риск кита;
- получить спортивные результаты;
- обработать апелляции;
- построить payout manifest;
- выдать Merkle proof;
- автоматически выплатить выигрыш;
- восстановиться после сбоя;
- показать пользователю понятный статус.

При этом backend не должен иметь бесконтрольный доступ к пользовательским деньгам. Деньги защищают смарт-контракты. Backend готовит данные и отправляет транзакции только в рамках разрешенных правил.

---

## 4. Что backend делает, а что не делает

Backend делает:

- расчеты;
- проверки;
- хранение данных;
- подготовку транзакций;
- мониторинг;
- админские workflow (рабочие процессы);
- интеграции с внешними API;
- построение Merkle proof;
- управление очередями выплат.

Backend не должен:

- менять правила тиража после Open;
- менять сумму выплаты после публикации root;
- выводить пользовательские средства;
- обходить KYC/compliance;
- скрывать ручные решения;
- использовать один API как единственный источник результата для денежных тиражей;
- отправлять claimFor без проверки proof и claimed-статуса.

---

## 5. Главные риски

Основные риски backend:

- потеря Merkle proof;
- неправильный расчет победителей;
- ошибка при сопоставлении спортивных событий;
- сбой индексатора;
- повторная отправка выплаты;
- компрометация executor wallet;
- исчерпание API quota;
- сбой RPC;
- потеря audit log;
- расхождение базы данных и блокчейна;
- слишком широкие права администратора.

Эти риски закрываются:

- резервным хранением manifest/proof;
- индексатором блокчейн-событий;
- идемпотентными jobs (повторный запуск не ломает состояние);
- очередью выплат;
- проверкой claimed-статуса перед claimFor;
- мониторингом executor wallet;
- multisig для критических действий;
- audit log минимум 5 лет, если это подтвердит юридический этап.

---

## 6. Главный вывод

Этап 9 - это технический фундамент между смарт-контрактами и интерфейсом.

Правильная серверная архитектура должна быть:

- надежной;
- проверяемой;
- устойчивой к сбоям;
- понятной для аудита;
- готовой к автоматическим выплатам;
- не опасной для пользовательских средств.

Главный принцип:

```text
backend может считать и обслуживать,
но не должен иметь возможность украсть или незаметно изменить деньги пользователя.
```
