# OX - Этап 10. Пользовательский интерфейс

Дата: 09.06.2026  
Версия: v1  
Проект: OX, спортивный экспресс  
Стартовый продукт: Суперэкспресс 15  
Будущий продукт: Мегаэкспресс 16  
Девиз: Честно. Прозрачно. Мгновенно.

---

## 1. Назначение этапа

Этап 10 описывает User Interface (пользовательский интерфейс) OX: публичный сайт/приложение, личный кабинет пользователя и основные экраны взаимодействия с продуктом.

Этот этап отвечает на вопросы:

- какие экраны нужны пользователю в MVP;
- как пользователь покупает билет;
- как отображаются 15 событий Суперэкспресса;
- как заложить будущий Мегаэкспресс 16;
- как показывать статистику и вероятности;
- как отображать системные ставки;
- как показывать статус тиража, результата и выплаты;
- как подключается wallet (кошелек);
- как работает KYC (верификация клиента);
- какие языки и локализация нужны;
- как интерфейс связан с backend API и смарт-контрактами;
- какие требования к мобильной версии, доступности и скорости.

Связь с предыдущими этапами:

- Этап 1 определяет ядро продукта.
- Этап 2 определяет жизненный цикл тиража.
- Этап 3 определяет билеты и системные ставки.
- Этап 4 определяет призовой фонд и джекпот.
- Этап 5 определяет защиту от кита.
- Этап 6 определяет автоматические выплаты.
- Этап 7 определяет спортивные результаты.
- Этап 8 определяет смарт-контракты.
- Этап 9 определяет backend API, indexer и payout executor.

Главное правило:

```text
интерфейс не должен усложнять продукт для пользователя, но обязан честно показывать правила, деньги, статус и риски.
```

---

## 2. Продуктовая цель интерфейса

OX должен ощущаться как простой спортивный экспресс с прозрачными правилами и автоматической выплатой.

Основной пользовательский сценарий MVP:

```text
зашел -> увидел текущий Суперэкспресс 15 -> выбрал исходы -> увидел стоимость -> купил билет -> дождался результата -> получил выплату
```

Интерфейс должен решать три задачи:

1. Помочь пользователю быстро понять текущий тираж.
2. Дать удобный выбор исходов и системных ставок.
3. Показать прозрачный статус денег, результата и выплаты.

## 4. MVP scope

В MVP входит:

- один активный продукт: Суперэкспресс 15;
- подготовка архитектуры интерфейса под Мегаэкспресс 16;
- список 15 спортивных событий;
- выбор исходов по каждому событию;
- одиночный полный билет 15/15;
- системные ставки, если они согласованы к MVP;
- расчет стоимости до оплаты;
- отображение джекпота и призового фонда;
- подключение wallet;
- покупка билета за USDC или другой согласованный токен;
- история билетов;
- статусы тиража;
- результаты событий;
- статус выплаты;
- блок статистики и вероятностей;
- базовая локализация;
- адаптивная мобильная версия;
- раздел правил и прозрачности.

В MVP не входит как обязательное:

- полноценная биржа ставок;
- live betting (ставки в реальном времени);
- социальная лента;
- внутренний чат;
- NFT-элементы;
- токен OX;
- сложная геймификация;
- кастомные аватары;
- публичные профили игроков.

Причина:

```text
MVP должен доказать главный цикл: тираж -> билет -> результат -> выплата.
```

## 5. Главный экран текущего тиража

Главный экран должен сразу показывать:

- название продукта: Суперэкспресс 15;
- девиз: Честно. Прозрачно. Мгновенно.;
- активный статус тиража;
- время до закрытия приема билетов;
- текущий джекпот;
- общий призовой фонд;
- стоимость одной комбинации;
- количество событий: 15;
- кнопку перехода к выбору исходов;
- блок "мои активные билеты", если wallet подключен.

Статусы тиража:

```text
Upcoming
Open
Locked
ResultsUnderReview
AppealActive
SettlementPending
PayoutsPending
PayoutsActive
Finished
Cancelled
RefundReady
```

Пользовательские формулировки:

| Технический статус | Текст для пользователя |
|---|---|
| Upcoming | Тираж скоро откроется |
| Open | Можно купить билет |
| Locked | Прием билетов закрыт |
| ResultsUnderReview | Результаты проверяются |
| AppealActive | Идет окно проверки результата |
| SettlementPending | Рассчитываем выигрыши |
| PayoutsPending | Готовим выплаты |
| PayoutsActive | Выплаты активны |
| Finished | Тираж завершен |
| Cancelled | Тираж отменен |
| RefundReady | Доступен возврат |

## 6. Экран событий

Список событий должен показывать 15 событий Суперэкспресса.

Для каждого события:

- номер события;
- вид спорта;
- турнир/лига;
- команды/участники;
- дата и время начала;
- статус события;
- варианты исходов;
- вероятность/проценты по исходам;
- источник статистики;
- отметка, если событие заменено или признано void.

Так как проект называется спортивный экспресс, а не футбольный экспресс, интерфейс должен поддерживать разные sports (виды спорта):

- футбол как основной стартовый вид;
- хоккей как возможное расширение;
- баскетбол как возможное расширение;
- другие виды спорта после отдельного согласования.

Правило:

```text
UI не должен жестко зашивать football-only логику.
```

## 7. Выбор исходов

Для MVP базовый формат исходов:

```text
1 / X / 2
```

Расшифровка:

- `1` - победа первой команды;
- `X` - ничья, если она применима к виду спорта и рынку;
- `2` - победа второй команды.

Для видов спорта без ничьей интерфейс должен поддерживать market template (шаблон рынка), где исходов может быть 2 или больше.

Требования:

- выбранный исход должен быть визуально понятен;
- нельзя купить билет с неполным набором обязательных исходов;
- пользователь должен видеть количество выбранных исходов;
- при системной ставке пользователь должен видеть число комбинаций;
- при изменении выбора цена пересчитывается сразу;
- недоступное событие нельзя выбрать;
- закрытое событие нельзя выбрать;
- если событие стало void, пользователь должен видеть причину.

## 8. System Bets

System Bets (системные ставки) должны быть понятны без математической перегрузки.

Интерфейс должен показывать:

- выбранную систему;
- сколько исходов выбрано;
- сколько комбинаций формируется;
- стоимость одной комбинации;
- итоговую стоимость билета;
- какие комбинации участвуют в билете;
- предупреждение, если система резко увеличивает цену.

Минимальные элементы:

```text
Выбор исходов
Количество комбинаций
Цена за комбинацию
Итоговая цена
Подтверждение покупки
```

UX-правило:

```text
пользователь должен понимать итоговую цену до подключения оплаты.
```

## 9. Ticket Builder

Ticket Builder (сборщик билета) должен быть центральным рабочим элементом интерфейса.

Он должен показывать:

- выбранные исходы;
- незаполненные события;
- системный режим;
- количество комбинаций;
- итоговую цену;
- предупреждения;
- кнопку покупки.

Состояния:

```text
Empty
PartiallySelected
ReadyToBuy
PriceChanged
DrawLocked
WalletRequired
KYCRequired
InsufficientBalance
AwaitingTransaction
Purchased
Failed
```

Важные правила:

- если comboPrice изменился до покупки, пользователь должен подтвердить новую цену;
- если тираж перешел в Locked, покупка блокируется;
- если баланс недостаточен, нужно показать понятную ошибку;
- если нужна KYC, нужно объяснить причину и следующий шаг;
- после покупки пользователь должен получить ticketId и статус.

## 10. Checkout и оплата

Checkout (экран оплаты) должен показывать:

- выбранный продукт;
- номер тиража;
- количество событий;
- количество комбинаций;
- цену за комбинацию;
- итоговую сумму;
- token (токен оплаты);
- network (сеть);
- wallet address;
- предупреждение о неизменности покупки после подтверждения;
- кнопку подтверждения.

После покупки:

- показывается txHash;
- показывается ticketId;
- билет появляется в My Tickets;
- пользователь видит статус Purchased;
- при ошибке показывается причина и безопасный повтор.

Важно:

```text
пользователь не должен платить gas за выплату выигрыша, если активен режим automatic payout.
```

При покупке билета пользователь может платить network fee (сетевую комиссию), если архитектура оплаты этого требует. Это отдельно уточняется на этапе выбора сети и кошелька.

## 11. Wallet connection

Wallet connection (подключение кошелька) нужно сделать максимально простым.

MVP должен поддерживать:

- подключение wallet;
- отображение сокращенного адреса;
- проверку network;
- переключение network, если возможно;
- отображение баланса;
- подпись сообщений, если нужна авторизация;
- disconnect (отключение кошелька).

Возможные подходы:

```text
WalletConnect
MetaMask
embedded wallet / social login - после отдельного решения
```

Рекомендация для MVP:

```text
начать с обычного wallet connection, а embedded wallet рассмотреть как отдельное UX-улучшение.
```

Почему:

- обычный wallet быстрее запустить;
- меньше юридических и кастодиальных рисков;
- embedded wallet может сильно улучшить массовый вход, но требует отдельной оценки безопасности.

## 12. My Tickets

Раздел My Tickets (мои билеты) должен показывать:

- активные билеты;
- завершенные билеты;
- отмененные/возвратные билеты;
- выигрышные билеты;
- билеты, ожидающие выплату;
- билеты, требующие KYC;
- фильтры по статусу;
- переход в детали билета.

Карточка билета:

- drawId;
- ticketId;
- дата покупки;
- сумма покупки;
- количество комбинаций;
- статус тиража;
- текущий результат;
- потенциальный/фактический выигрыш, если известен;
- статус выплаты.

```text
Возврат выплачен
```

## 14. Results screen

Экран результатов должен показывать:

- финальные результаты 15 событий;
- источник результата;
- статус проверки;
- были ли апелляции;
- были ли void events;
- сколько событий признано валидными;
- активен ли jackpot;
- какие уровни выплат сработали;
- ссылку на публичный отчет.

Правила:

- нельзя показывать результат как финальный до FinalConfirmed;
- при спорном результате показывается статус проверки;
- ручной fallback должен быть отмечен в отчете;
- пользователь должен понимать, почему событие void или почему включен refund.

## 15. Payout Status

Интерфейс выплат должен отображать:

- ожидается ли автоматическая выплата;
- была ли выплата отправлена;
- txHash выплаты;
- сумма выплаты;
- token;
- статус KYC;
- доступен ли manual claim;
- причина задержки, если задержка есть.

Статусы:

```text
PayoutsPending
AutoPayoutQueued
AutoPayoutProcessing
Paid
ManualClaimAvailable
BlockedKYC
GasPaused
FailedRetryable
FailedFinal
RefundReady
RefundPaid
```

Пользовательские тексты:

| Статус | Текст |
|---|---|
| AutoPayoutQueued | Выплата поставлена в очередь |
| AutoPayoutProcessing | Отправляем выплату |
| Paid | Выплата отправлена |
| ManualClaimAvailable | Можно получить вручную |
| BlockedKYC | Нужна верификация |
| GasPaused | Автовыплаты временно задержаны |
| RefundReady | Доступен возврат |

Важно:

```text
GasPaused не должен звучать как потеря денег. Нужно объяснить, что право на выплату сохраняется.
```

## 16. KYC flow

KYC (верификация клиента) должен быть встроен без лишнего давления на пользователя.

Сумма, с которой начинается обязательная KYC, пока не фиксируется:

```text
KYC threshold = TBD
```

TBD (to be determined) означает: будет определено позже на юридическом и комплаенс-этапе.

Интерфейс должен поддерживать:

- NotRequired;
- Required;
- Pending;
- Approved;
- Rejected;
- Expired.

Где показывать KYC:

- в Account;
- в Checkout, если KYC нужен до покупки;
- в Payout Status, если KYC нужен до выплаты;
- в Ticket Details, если выплата заблокирована.

Правило:

```text
если KYC блокирует выплату, пользователь должен видеть причину и следующий шаг.
```

## 17. Статистика и вероятности

Этот блок уже согласован как важная фича.

На первом этапе интерфейс показывает probabilities (вероятности) и проценты на основе внешних спортивных/букмекерских API.

На втором этапе можно добавить real stake distribution (реальное распределение ставок пользователей), если это не создает риск манипуляций и не помогает китам.

MVP показывает:

- процент по каждому исходу;
- источник данных;
- время последнего обновления;
- дисклеймер, что проценты не гарантируют результат;
- отсутствие процента, если источник недоступен.

Запрещено:

- выдавать вероятность как гарантию;
- скрывать источник;
- показывать устаревшие данные без отметки;
- раскрывать внутренние risk-параметры, которые помогают киту.

## 18. Локализация

Интерфейс должен быть готов к localization (локализации) на языки целевых стран.

Приоритетные языки:

- русский;
- английский;
- испанский;
- португальский;
- индонезийский;
- вьетнамский;
- узбекский;
- казахский;
- суахили.

Требования:

- весь пользовательский текст через translation keys (ключи перевода);
- даты и время с учетом timezone (часового пояса);
- суммы с понятным форматированием;
- длинные слова не должны ломать layout;
- юридические тексты переводятся отдельно и проверяются;
- технические английские термины в русской версии объясняются в скобках.

Пример:

```text
Merkle proof (доказательство выплаты)
Gas (комиссия сети)
Wallet (кошелек)
Payout (выплата)
```

## 19. Public Transparency Report

Публичный отчет нужен для доверия.

Он должен показывать:

- drawId;
- правила тиража;
- список событий;
- финальные результаты;
- jackpot;
- prize pool;
- количество билетов;
- payoutRoot;
- resultRoot;
- manifestHash;
- txHash публикации root;
- manual fallback markers, если были;
- audit summary без персональных данных.

Пользовательский вариант отчета должен быть простым.

Технический вариант отчета может содержать:

- root/hash;
- txHash;
- blockNumber;
- contract address;
- proof verification link.

## 20. Help / Rules

Раздел Help / Rules (помощь/правила) должен объяснять:

- что такое Суперэкспресс 15;
- как выбрать исходы;
- как считается билет;
- что такое системная ставка;
- как работает джекпот;
- что происходит при void events;
- когда бывает refund;
- как работает автоматическая выплата;
- когда может потребоваться KYC;
- что делать, если выплата задержалась;
- почему OX стремится к децентрализации поэтапно.

Правило:

```text
тексты помощи должны быть короткими и понятными, без юридического перегруза.
```

## 21. Account

Account (аккаунт) должен включать:

- wallet address;
- network;
- language;
- KYC status;
- notification preferences;
- linked email/telegram, если будет принято;
- privacy settings;
- export history, если потребуется.

MVP может работать без классической регистрации, через wallet-based account (аккаунт на основе кошелька).

Решение:

```text
MVP: wallet-first account.
Email/Telegram notifications: опционально после отдельного решения.
```

## 22. Notifications

Уведомления в MVP можно разделить:

- in-app notifications (уведомления внутри интерфейса);
- optional email/telegram notifications после отдельного решения.

События для уведомлений:

- тираж открыт;
- билет куплен;
- прием билетов закрыт;
- результаты проверяются;
- результаты финализированы;
- выигрыш найден;
- выплата отправлена;
- нужен KYC;
- доступен refund;
- произошла задержка автовыплаты.

## 23. Адаптивность и mobile-first

Основная аудитория, скорее всего, будет часто заходить с телефона. Поэтому MVP должен быть mobile-first.

Требования:

- интерфейс работает на мобильных экранах;
- выбор исходов удобен пальцем;
- кнопки не слишком мелкие;
- ticket builder не перекрывает события;
- длинные названия команд не ломают карточки;
- checkout помещается на мобильном экране;
- статус выплаты понятен без горизонтальной прокрутки;
- скорость загрузки приемлема на слабом интернете.

Desktop-версия также нужна, но мобильная версия является приоритетом.

---

## 24. Доступность

Accessibility (доступность) нужна не только для формальных требований, но и для качества продукта.

Минимум:

- достаточный contrast (контраст);
- понятные focus states (состояния фокуса);
- управление клавиатурой;
- aria labels для важных кнопок;
- ошибки форм текстом, не только цветом;
- адаптация под screen readers, где применимо.

---

## 25. UI states

```text
Loading

Ready

Offline
StaleData

Unauthorized
```

Особенно важно:

- при ошибке API объяснять, что делать;
- при wrong network предлагать переключение;
- при stale data показывать время последнего обновления.

## 27. Интеграция со смарт-контрактами

UI не должен самостоятельно считать критические финансовые итоги.

UI может:

- читать данные контракта;
- инициировать покупку билета;
- показывать txHash;
- открывать explorer link;
- показывать proof/status;
- давать manual claim, если он доступен.

UI не должен:

- менять payout amount;
- менять recipient;
- публиковать root;
- запускать settlement;
- обходить KYC;
- скрывать статус контракта.

## 28. Design principles

Визуально OX должен быть:

- современным;
- понятным;
- надежным;
- не похожим на перегруженную букмекерскую витрину;
- не похожим на казино;
- с акцентом на прозрачность и статус денег.

Рекомендации:

- главный экран сразу показывает текущий тираж;
- не делать маркетинговую посадочную страницу вместо продукта;
- не перегружать первый экран вторичными блоками;
- деньги и статусы показывать ясно;
- правила показывать коротко рядом с действиями;
- техническую прозрачность вынести в отдельный отчет.

## 29. Ошибки и предупреждения

Типовые ошибки:

| Ошибка | Что показать |
|---|---|
| WalletDisconnected | Подключите кошелек |
| WrongNetwork | Переключите сеть |
| InsufficientBalance | Недостаточно средств |
| DrawLocked | Прием билетов закрыт |
| PriceChanged | Цена изменилась, подтвердите новую |
| KYCRequired | Требуется верификация |
| GasPaused | Автовыплаты временно задержаны, право на выплату сохраняется |
| APIUnavailable | Данные временно недоступны |
| TransactionRejected | Транзакция отклонена в кошельке |
| TransactionFailed | Транзакция не прошла |

Правило:

```text
ошибка должна объяснять причину и следующий шаг.
```

## 31. Performance

Требования:

- быстрый первый экран;
- кеширование справочных данных;
- отдельная загрузка тяжелых отчетов;
- polling или websocket для статусов после отдельного решения;
- отсутствие блокировки интерфейса при ожидании транзакции;
- graceful degradation (мягкое снижение функциональности) при сбое части API.

Предварительный ориентир:

```text
Current Draw должен открываться быстро даже на мобильном интернете.
```

Конкретные метрики производительности фиксируются на frontend development этапе.

## 32. Security UX

Интерфейс должен помогать пользователю не ошибиться.

Требования:

- показывать адрес кошелька перед оплатой;
- показывать сеть;
- показывать токен оплаты;
- не просить seed phrase;
- не просить приватный ключ;
- предупреждать о фишинговых рисках в help-разделе;
- использовать официальные wallet-подключения;
- показывать только проверенные contract addresses из конфигурации.

## 35. Итоговое решение

Для MVP принимается пользовательский интерфейс:

```text
mobile-first web interface для Суперэкспресса 15 с подготовкой под Мегаэкспресс 16
```

Ключевые решения:

- главный экран сразу ведет в текущий тираж;
- продукт называется спортивный экспресс, не футбольный экспресс;
- Суперэкспресс 15 является стартовым продуктом;
- Мегаэкспресс 16 закладывается архитектурно на второй этап;
- пользователь видит джекпот, призовой фонд, цену комбинации и статус;
- Ticket Builder является центральным элементом;
- статистика и вероятности показываются в MVP;
- реальное распределение ставок переносится после MVP;
- автоматические выплаты отображаются простыми статусами;
- KYC threshold пока TBD;
- локализация закладывается сразу;
- публичная прозрачность отделяется от основного простого сценария.

Главный принцип:

```text
интерфейс должен делать OX простым для пользователя, не скрывая правила, деньги и статус выплаты.
```
