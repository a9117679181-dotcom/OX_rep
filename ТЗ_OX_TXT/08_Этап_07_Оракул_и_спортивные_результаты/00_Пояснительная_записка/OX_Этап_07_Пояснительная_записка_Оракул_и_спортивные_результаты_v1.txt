# OX - Этап 07. Пояснительная записка

## Оракул и спортивные результаты

Дата: 09.06.2026  
Версия: v1  
Проект: OX, спортивный экспресс  
Стартовый продукт: Суперэкспресс 15  
Девиз: Честно. Прозрачно. Мгновенно.

---

## 1. Что находится в этом этапе

Этот этап описывает, как OX будет получать, проверять и фиксировать результаты спортивных событий.

Для проекта это один из самых важных блоков. Если результат события будет определен неправильно, то дальше неправильно посчитаются победители, призовой фонд, джекпот и выплаты.

Поэтому результат нельзя брать из одного случайного источника. Нужна система, которая:

- получает данные из нескольких источников;
- сверяет результаты между собой;
- видит расхождения;
- останавливает спорные случаи;
- допускает ручное решение только как аварийный сценарий;
- сохраняет доказательства и логи;
- не запускает выплаты, пока результаты не прошли проверку.

---

## 2. Главное решение

Для MVP фиксируем гибридную архитектуру:

```text
3 sports data API + result aggregator + appeal window + controlled on-chain publication
```

Расшифровка:

- `sports data API (API спортивных данных)` - внешние сервисы, которые отдают расписание, статусы матчей и финальные результаты.
- `result aggregator (агрегатор результатов)` - серверный модуль OX, который собирает данные из источников и сравнивает их.
- `appeal window (окно апелляции)` - время, когда результат уже показан пользователям, но выплаты еще не запущены.
- `on-chain publication (публикация в блокчейне)` - фиксация финального результата или хэша результата для прозрачности.

Базовое правило:

```text
1 основной API + 2 резервных API
```

Основной API нужен для обычной работы. Два резервных API нужны не для красоты, а для проверки и аварийного переключения.

---

## 3. Почему не один API

Один API дешевле и проще, но это слабое место.

Риски одного источника:

- источник может ошибиться;
- источник может задержать финальный результат;
- матч может быть перенесен, прерван или отменен;
- один и тот же матч может иметь разные ID у разных провайдеров;
- может быть ошибка в часовых поясах;
- API может временно не работать;
- при споре будет трудно доказать, почему OX выбрал именно этот результат.

Поэтому для денежных тиражей нужен минимум трехисточниковый подход.

---

## 4. Что делаем с Chainlink

Chainlink Functions не закладываем как production-зависимость, потому что сервис уходит в sunset (вывод из эксплуатации) 1 сентября 2026 года.

Целевой путь для будущей более автономной версии:

```text
Chainlink CRE (Chainlink Runtime Environment, среда выполнения Chainlink)
```

CRE подходит лучше, потому что может запускать workflow (рабочий процесс): получить данные из API, обработать их, сформировать отчет и передать его в смарт-контракт.

Для MVP не делаем вид, что все сразу полностью децентрализовано. Правильнее идти поэтапно:

1. На MVP: серверный агрегатор OX + 3 API + ручной аварийный fallback + логи.
2. После тестов и аудита: подключение Chainlink CRE для более независимой публикации результата.
3. После стабилизации: максимальная автоматизация и снижение роли администратора.

---

## 5. Как определяется результат

Для стартового продукта `Суперэкспресс 15` основной тип исхода:

```text
1 / X / 2
```

То есть:

- `1` - победа первой команды;
- `X` - ничья;
- `2` - победа второй команды.

Для футбола результат считается по основному времени матча, включая добавленное арбитром время, но без дополнительного времени и серии пенальти, если в карточке события заранее не указано другое правило.

Это нужно обязательно показывать пользователю до покупки билета.

---

## 6. Что делать со спорными событиями

Если событие отменено, перенесено, прервано или источники дают разные результаты, система не должна молча считать выплату.

Базовая логика:

- 1-2 проблемных события: тираж может быть сокращен до 14 или 13 событий по правилам этапа 4.
- 3 и более проблемных событий: тираж признается несостоявшимся, ставки возвращаются.
- если результат спорный, включается ручная проверка;
- если решение влияет на выплаты, оно подтверждается через multisig (мультиподпись).

---

## 7. Статистика и проценты вероятностей

На первом этапе можно показывать проценты вероятностей из букмекерских котировок или API-данных, если лицензия это разрешает.

Важно: эти проценты не являются результатом OX и не участвуют в расчете выигрышей. Это только информационный слой для пользователя.

На втором этапе можно добавить реальное распределение ставок пользователей OX, но осторожно. Если показывать его слишком рано, крупные игроки смогут видеть поведение толпы и использовать это против экономики проекта. Поэтому предварительное решение: показывать агрегированное распределение после закрытия продаж или с задержкой.

---

## 8. Главный вывод

Этап 7 фиксирует правильный подход:

```text
результаты должны быть быстрыми, но не ценой ошибки
```

OX должен стремиться к мгновенности, но сначала нужна надежность. Выплаты можно запускать только после того, как результат подтвержден, проверен и прошел окно апелляции.

Это напрямую поддерживает девиз проекта:

```text
Честно. Прозрачно. Мгновенно.
```

Но порядок здесь такой:

```text
сначала честно и прозрачно, затем мгновенно.
```
