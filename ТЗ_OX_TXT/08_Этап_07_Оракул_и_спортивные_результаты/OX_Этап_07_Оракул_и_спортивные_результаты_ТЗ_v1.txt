# OX - Этап 07. Оракул и спортивные результаты

Дата: 09.06.2026  
Версия: v1  
Проект: OX, спортивный экспресс  
Стартовый продукт: Суперэкспресс 15  
Будущий продукт: Мегаэкспресс 16  
Девиз: Честно. Прозрачно. Мгновенно.

---

## 1. Назначение этапа

Этап 7 описывает архитектуру получения, проверки, публикации и оспаривания спортивных результатов в OX.

Этот этап отвечает на вопросы:

- какие источники спортивных данных используются;
- как система выбирает 15 событий для тиража;
- как события связываются с внешними API;
- как определяется финальный результат;
- что делать при расхождении источников;
- как работает ручной fallback (ручной резервный сценарий);
- как результат проходит окно апелляции;
- что публикуется в блокчейн;
- как результат связан с расчетом победителей и выплатами.

Связь с предыдущими этапами:

- Этап 2 определяет жизненный цикл тиража, окно апелляции 6 часов и payout timelock (задержку выплат) 6 часов.
- Этап 3 определяет билеты и системные ставки.
- Этап 4 определяет сокращенные тиражи 14/13 и refund (возврат) при 3+ отменах.
- Этап 5 определяет защиту от кита и риск-движок.
- Этап 6 определяет автоматические выплаты после финализации результатов.

Главное правило:

```text
выплаты нельзя запускать, пока спортивные результаты не подтверждены и не прошли окно апелляции.
```

## 2. Термины

**Oracle (оракул)** - механизм, который переносит внешние данные в систему OX и, при необходимости, в блокчейн.

**Sports data API (API спортивных данных)** - внешний источник расписания, статусов матчей, финальных счетов и статистики.

**Primary API (основной API)** - главный источник данных для обычной работы.

**Backup API (резервный API)** - дополнительный источник для проверки и аварийного переключения.

**Result aggregator (агрегатор результатов)** - серверный модуль OX, который получает данные из API, нормализует их и сравнивает между собой.

**Result consensus (согласование результата)** - правило, по которому система решает, что результат достаточно надежен.

**Manual fallback (ручной резервный сценарий)** - процедура ручного ввода или подтверждения результата при сбое автоматических источников.

**Appeal window (окно апелляции)** - период, когда результат опубликован для пользователей, но выплаты еще не запущены.

**Result root (корень результатов)** - хэш или Merkle root (меркл-корень) финального списка результатов тиража.

**Event mapping (сопоставление события)** - связь внутреннего события OX с ID этого же события у разных провайдеров.

**Final result (финальный результат)** - результат события, который прошел проверку и может участвовать в расчете билетов.

**Void event (аннулированное событие)** - событие, которое исключено из расчета.

**Valid events count (количество валидных событий)** - фактическое число событий, участвующих в расчете: 15, 14 или 13.

**CRE (Chainlink Runtime Environment, среда выполнения Chainlink)** - целевая технология Chainlink для workflow (рабочих процессов), которая может получать API-данные, выполнять логику и передавать отчет в смарт-контракт.

## 3. Внешние технологические факты на 09.06.2026

На дату подготовки этапа 7 проверены следующие внешние факты:

- Chainlink Functions уходит в sunset (вывод из эксплуатации) 1 сентября 2026 года. Поэтому Chainlink Functions нельзя закладывать как production-зависимость.
- Chainlink рекомендует миграцию к Chainlink CRE.
- Chainlink CRE поддерживает workflow (рабочие процессы), HTTP/API calls (запросы к API), cron trigger (запуск по расписанию), on-chain writes (запись в блокчейн) и monitoring (мониторинг).
- API-Football/API-Sports может использоваться как стартовый источник футбольной статистики и тестовых данных. На дату проверки тариф Pro указан как 19 USD в месяц и 7 500 запросов в день.
- TheSportsDB может рассматриваться как дешевый резервный источник или источник для тестов.
- SportsDataIO может рассматриваться как более профессиональный/enterprise-кандидат, но требует отдельной проверки договора, SLA и стоимости.

Важно:

```text
конкретные production-провайдеры для денежных тиражей утверждаются только после проверки лицензии, качества данных, покрытия стран, SLA и права использовать данные в коммерческом продукте.
```

## 4. Архитектурное решение MVP

Для MVP фиксируется гибридная архитектура:

```text
primary API + backup API 1 + backup API 2
        -> result aggregator
        -> result consensus
        -> Results Under Review
        -> appeal window
        -> result finalization
        -> settlement engine
        -> payout system
```

Роли компонентов:

| Компонент | Назначение |
|---|---|
| Primary API | основной источник расписания и результатов |
| Backup API 1 | резервный источник для проверки результата |
| Backup API 2 | второй резервный источник для проверки результата |
| Result aggregator | сбор, нормализация и сравнение данных |
| Admin panel | контроль спорных событий и ручной fallback |
| Result registry | хранение финального результата и версий |
| Smart contract | фиксация финального result root или hash |
| Settlement engine | расчет победителей после финализации |

Для MVP допускается, что result aggregator работает как серверный модуль OX. Это честнее, чем заявлять полную децентрализацию до готовности CRE, аудита и юридической проверки провайдеров.

Целевой путь после MVP:

```text
Chainlink CRE workflow -> ResultOracleReceiver -> on-chain result root
```

## 5. Почему не используем один источник

Вариант 1: один API.

Плюсы:

- дешевле;
- быстрее реализовать;
- меньше интеграций.

Минусы:

- один источник ошибки;
- нет независимой сверки;
- сложнее защищаться при споре;
- выше риск неверного расчета;
- слабее доверие пользователей.

Вывод: подходит только для прототипа и тестов, но не как production-модель денежных тиражей.

Вариант 2: три API.

Плюсы:

- можно сравнивать результаты;
- можно пережить отказ одного источника;
- можно видеть расхождения;
- повышается доверие;
- есть база для автоматического consensus (согласования).

Минусы:

- дороже;
- сложнее mapping (сопоставление событий);
- нужно следить за лимитами и лицензиями;
- больше технической поддержки.

Вывод: принимается для MVP денежных тиражей как правильный баланс надежности и сложности.

## 6. Провайдеры данных

### 6.1. Принцип выбора

На этапе 7 не фиксируем жесткий каскад конкретных провайдеров.

Причина:

- цены могут меняться;
- лицензии могут запрещать коммерческое использование;
- качество данных нужно проверять на реальных матчах;
- покрытие стран и лиг отличается;
- не все провайдеры одинаково хорошо обрабатывают отмены, переносы, технические поражения, дополнительное время и пенальти.

Фиксируем не названия поставщиков, а обязательную модель:

```text
1 основной provider + 2 резервных provider
```

### 6.2. Кандидаты на проверку

| Кандидат | Предварительная роль | Комментарий |
|---|---|---|
| API-Football / API-Sports | dev/test, стартовая статистика, возможный primary после проверки | Низкий входной бюджет, подходит для ранней интеграции |
| TheSportsDB | дешевый backup или dev/test | Нужна проверка качества и задержек |
| SportsDataIO | production/enterprise кандидат | Нужны переговоры, SLA, договор и цена |
| Official league/federation feeds | источник повышенного доверия | Дороже и сложнее в подключении |
| Odds provider / bookmaker odds API | вероятности и котировки | Не должен быть источником финального результата без отдельной проверки |

### 6.3. Правило production-допуска

Провайдер допускается к production только если:

- договор разрешает коммерческое использование;
- покрытие включает нужные страны и лиги;
- есть стабильные IDs событий;
- есть финальные результаты и статусы отмены/переноса;
- есть история изменений результата;
- есть понятные лимиты запросов;
- есть SLA или понятная операционная надежность;
- есть техническая документация;
- есть возможность резервной интеграции.

## 7. Формирование списка событий

### 7.1. MVP

В MVP список событий формирует администратор.

Порядок:

1. Система загружает потенциальные события из API.
2. Администратор выбирает 15 событий для `Суперэкспресса 15`.
3. Система проверяет, что у каждого события есть mapping минимум в primary API.
4. Желательно, чтобы у каждого события был mapping минимум в двух источниках.
5. Перед публикацией тиража администратор проверяет карточки событий.
6. После открытия продаж событие нельзя заменить другим событием.

### 7.2. После MVP

После MVP можно добавить Cron-job (автоматическую задачу по расписанию), который:

- собирает события;
- фильтрует по времени начала;
- исключает рискованные лиги;
- проверяет наличие mapping у трех источников;
- готовит черновик тиража;
- передает черновик администратору на подтверждение.

Полностью автоматический выбор событий допускается только после накопления статистики по ошибкам, отменам, API-задержкам и качеству лиг.

## 8. Требования к событию

Минимальные поля события OX:

```text
eventId
drawId
sportType
leagueName
country
homeTeamName
awayTeamName
scheduledStartUtc
marketType
allowedOutcomes
eventStatus
providerMappings[]
resultRule
createdBy
createdAt
lockedAt
```

Для стартового продукта:

```text
sportType = football
marketType = 1X2
allowedOutcomes = 1, X, 2
resultRule = fullTimeRegularOnly
```

`fullTimeRegularOnly (только основное время)` означает:

- учитывается основное время матча;
- учитывается добавленное арбитром время;
- не учитывается дополнительное время;
- не учитывается серия пенальти;
- исключение допускается только если правило явно указано в карточке события до открытия продаж.

Для будущих видов спорта нужно создавать отдельный `resultRule`. Например, хоккей по периодам или исход матча с учетом овертайма нельзя смешивать с футбольным правилом 1X2.

## 9. Event mapping

Event mapping (сопоставление события) является критическим требованием.

У одного и того же матча разные провайдеры могут иметь разные:

- ID матча;
- названия команд;
- названия лиги;
- время начала;
- статус;
- формат результата.

Структура mapping:

```text
providerName
providerEventId
providerLeagueId
providerHomeTeamId
providerAwayTeamId
providerStartTimeUtc
mappingConfidence
mappingStatus
lastVerifiedAt
```

Статусы mapping:

| Статус | Значение |
|---|---|
| Mapped | сопоставление подтверждено |
| Candidate | найден возможный матч, нужна проверка |
| Ambiguous | найдено несколько похожих матчей |
| Missing | матч не найден у провайдера |
| Rejected | сопоставление отклонено |

Правило:

```text
если mapping ambiguous или missing у двух и более источников, событие нельзя открывать в продаже без ручного подтверждения.
```

## 10. Статусы события

| Статус | Значение |
|---|---|
| Draft | событие в черновике |
| Approved | событие утверждено для тиража |
| Locked | событие зафиксировано после открытия продаж |
| Scheduled | ожидает начала |
| Live | событие идет |
| FinishedPending | провайдер сообщил завершение, идет проверка |
| FinalCandidate | результат предварительно подтвержден источниками |
| SourceMismatch | источники расходятся |
| ManualReview | нужна ручная проверка |
| FinalConfirmed | результат подтвержден |
| Postponed | событие перенесено |
| Cancelled | событие отменено |
| Abandoned | событие прервано |
| Void | событие исключено из расчета |

## 11. Получение результатов

Result polling (опрос результатов) работает по расписанию.

Предварительные интервалы MVP:

| Период | Частота |
|---|---:|
| За 24 часа до старта | каждые 6 часов |
| За 3 часа до старта | каждый час |
| Во время события | каждые 1-5 минут, если тариф API позволяет |
| После ожидаемого окончания | каждые 5 минут до статуса finished |
| После первого final | контрольные проверки через 15, 30 и 60 минут |
| После публикации результата | мониторинг изменений минимум 12 часов |

Если API имеет жесткие лимиты, частота снижается, но финальный результат должен быть проверен до запуска расчета.

---

## 12. Нормализация результата

Каждый provider может отдавать данные в разном формате.

Result aggregator должен привести их к единой структуре:

```text
providerName
providerEventId
providerStatus
providerUpdatedAt
homeScoreRegular
awayScoreRegular
homeScoreExtraTime
awayScoreExtraTime
penaltyHomeScore
penaltyAwayScore
resultOutcome
resultConfidence
rawPayloadHash
rawPayloadStorageRef
```

Для футбольного 1X2:

```text
if homeScoreRegular > awayScoreRegular -> 1
if homeScoreRegular == awayScoreRegular -> X
if homeScoreRegular < awayScoreRegular -> 2
```

Если provider отдает только итог после дополнительного времени или пенальти, а регулярный счет недоступен, событие переводится в ManualReview.

## 13. Consensus rules

### 13.1. Автоматическое подтверждение

Результат может стать `FinalCandidate`, если:

- primary API и хотя бы один backup API совпали по outcome;
- или оба backup API совпали, а primary API недоступен, но событие дополнительно помечено как `NeedsOperatorCheck`;
- нет признаков отмены, переноса, технического поражения или спорного статуса;
- прошло минимальное время после окончания события.

Предварительный параметр:

```text
resultConfirmationDelay = 60 минут
```

### 13.2. Расхождение источников

Если источники расходятся:

| Ситуация | Действие |
|---|---|
| score разный, но outcome одинаковый | можно принять outcome, но оставить предупреждение в логах |
| outcome разный | SourceMismatch и ManualReview |
| primary отсутствует, backup 1 и backup 2 совпали | FinalCandidate с флагом NeedsOperatorCheck |
| доступен только один источник | ждать или ManualReview, но не запускать автоматический расчет |
| provider изменил результат после FinalCandidate | перезапустить проверку и открыть новую версию результата |

### 13.3. Финализация

`FinalConfirmed` допускается только после:

- прохождения consensus;
- публикации пользователям;
- завершения окна апелляции;
- отсутствия активных апелляций;
- или отклонения/закрытия апелляций.

## 14. Переносы, отмены и прерывания

Базовые правила MVP:

| Ситуация | Правило |
|---|---|
| Матч перенесен в пределах 48 часов | можно ждать результата, если это не ломает жизненный цикл тиража |
| Матч перенесен более чем на 48 часов | событие исключается из расчета |
| Матч отменен | событие исключается из расчета |
| Матч прерван и официально не завершен | ManualReview |
| Матч прерван, но официально засчитан | ManualReview с доказательством официального решения |
| Техническое поражение | ManualReview |
| Ошибка времени начала | ManualReview, если влияет на покупку билетов или закрытие продаж |

Связь с этапом 4:

```text
1-2 исключенных события -> валидный сокращенный тираж 14/13
3+ исключенных события -> void draw и refund 100%
```

После открытия продаж событие нельзя заменить другим событием. Его можно только подтвердить, исключить или перевести в спорный статус.

## 15. Results Under Review и апелляции

После получения предварительных финальных результатов тираж переводится в:

```text
Results Under Review
```

Пользователь видит:

- список событий;
- исходы;
- статус подтверждения;
- время начала и окончания окна апелляции;
- причину спорного статуса, если она есть.

Предварительное правило:

```text
appealWindow = 6 часов
payoutTimelock = 6 часов
```

Если апелляция подана:

- результат спорного события переводится в ManualReview;
- расчет победителей не запускается;
- оператор результатов проверяет источник;
- если решение влияет на выплаты, требуется multisig 2 из 3;
- после изменения результата публикуется новая версия;
- окно апелляции запускается заново.

Предварительный лимит:

```text
maxAppealRestarts = 2
```

Исключения допускаются только через multisig и с публичным логом причины.

## 16. Manual fallback

Manual fallback разрешен только как аварийная процедура.

Основания:

- primary API недоступен;
- два или три источника расходятся по outcome;
- событие перенесено, отменено или прервано;
- provider изменил результат после публикации;
- mapping оказался ошибочным;
- API вернул явно неверный статус;
- юридически значимый спор.

Процедура:

1. ResultOperator (оператор результатов) создает manual proposal (ручное предложение).
2. К proposal прикладываются доказательства: provider payload, официальная страница лиги, логи API, timestamp.
3. Система показывает, как решение повлияет на validEventsCount и выплаты.
4. Если решение влияет на расчет, требуется multisig 2 из 3.
5. После подтверждения создается новая версия результата.
6. Пользователи видят, что результат был подтвержден вручную.
7. Все действия хранятся в audit log.

Запрещено:

- менять результат без лога;
- менять результат после выплат без отдельной процедуры инцидента;
- использовать manual fallback как обычный режим;
- скрывать от пользователя факт ручного решения.

## 17. Публикация в блокчейн

В MVP не нужно публиковать каждое API-наблюдение в блокчейн. Это дорого и избыточно.

Публиковать нужно финальную проверяемую сводку:

```text
drawId
resultVersion
validEventsCount
resultRoot
resultManifestHash
finalizedAt
```

`resultManifest` хранится off-chain (вне блокчейна), но его hash/root публикуется on-chain (в блокчейне). Это позволяет доказать, что после финализации список результатов не был незаметно изменен.

Минимальные функции будущего контракта:

```text
publishResultRoot(drawId, resultVersion, resultRoot, manifestHash, validEventsCount)
challengeResultRoot(drawId, resultVersion, reasonCode)
finalizeResultRoot(drawId, resultVersion)
cancelResultRoot(drawId, resultVersion, reasonCode)
getResultRoot(drawId)
```

Критические изменения result root должны проходить через контролируемую процедуру и multisig.

## 18. Chainlink CRE после MVP

Целевая схема:

```text
CRE workflow
  -> fetch primary API
  -> fetch backup API 1
  -> fetch backup API 2
  -> normalize
  -> consensus
  -> generate report
  -> write report to ResultOracleReceiver
```

Преимущества:

- меньше доверия к серверу OX;
- workflow можно проверять;
- есть встроенный мониторинг исполнения;
- можно использовать cron trigger (запуск по расписанию);
- можно делать on-chain write (запись в блокчейн);
- логика ближе к децентрализованной модели.

Ограничения:

- CRE нужно отдельно протестировать;
- workflow должен быть детерминированным;
- зависимости должны быть совместимы с WASM;
- API-ключи должны храниться как secrets (секреты);
- нужно проверить поддерживаемые сети;
- требуется аудит workflow и receiver-контракта.

Решение:

```text
MVP не зависит от Chainlink Functions.
После MVP готовим миграцию result oracle flow на Chainlink CRE.
```

## 19. Статистика и вероятности

В интерфейсе можно показывать информационные проценты:

- вероятность победы первой команды;
- вероятность ничьей;
- вероятность победы второй команды.

### 19.1. MVP

На MVP источник вероятностей:

```text
bookmaker odds API или sports stats API
```

Если используются букмекерские odds (коэффициенты), нужно нормализовать implied probability (расчетную вероятность):

```text
rawP1 = 1 / odds1
rawPX = 1 / oddsX
rawP2 = 1 / odds2
sumRaw = rawP1 + rawPX + rawP2

P1 = rawP1 / sumRaw
PX = rawPX / sumRaw
P2 = rawP2 / sumRaw
```

Требования:

- проценты не участвуют в расчете выигрышей;
- проценты не являются обещанием результата;
- источник odds должен быть лицензирован;
- если данных нет, блок процентов скрывается;
- пользователь должен понимать, что это информационный показатель.

### 19.2. После MVP

Можно добавить реальное распределение ставок OX:

```text
сколько пользователей выбрали 1 / X / 2
```

Ограничение:

```text
не показывать точное распределение в реальном времени до закрытия продаж, если это помогает крупному игроку.
```

Предварительное решение:

- во время продаж показывать только внешние market probabilities (рыночные вероятности);
- после закрытия продаж показывать агрегированное распределение выборов пользователей;
- точный механизм отображения согласовать с этапом интерфейса и этапом защиты от кита.

## 26. Безопасность

Основные риски:

- подмена результата;
- компрометация API-ключей;
- ошибка mapping;
- один источник ошибся;
- оператор вручную ввел неверный результат;
- событие было изменено после открытия продаж;
- provider изменил результат после публикации;
- пользователь оспаривает результат после выплат;
- логов недостаточно для расследования.

Меры защиты:

- 3 источника данных;
- хранение raw payload hash;
- audit log всех действий;
- разграничение ролей;
- multisig для решений, влияющих на выплаты;
- запрет замены события после Open;
- версионирование result manifest;
- окно апелляции до выплат;
- запрет запуска settlement при SourceMismatch;
- резервное копирование logs и manifest;
- мониторинг provider health.

## 27. Роли и права

| Роль | Права |
|---|---|
| EventOperator | готовит список событий и mapping |
| ResultOperator | проверяет результаты и создает manual proposal |
| OracleAdmin | управляет provider configuration |
| MultisigSigner | подтверждает критические решения |
| Auditor | читает логи и отчеты |
| Support | видит пользовательские обращения и апелляции |

Разделение ролей:

- EventOperator не должен единолично финализировать результаты.
- ResultOperator не должен менять правила тиража.
- OracleAdmin не должен вручную менять результат без proposal.
- MultisigSigner не должен иметь доступ к API-ключам.
- Auditor должен иметь read-only доступ.

## 30. Критерии приемки

Этап считается реализованным, если:

- можно создать тираж на 15 событий;
- каждое событие имеет provider mapping;
- система получает результаты минимум из primary API;
- система поддерживает 2 backup API;
- результат нормализуется в 1/X/2;
- consensus rules работают;
- SourceMismatch блокирует settlement;
- ManualReview требует лога;
- критические ручные решения требуют multisig;
- работает окно апелляции;
- result manifest версионируется;
- result root/hash публикуется или готов к публикации;
- settlement engine получает только FinalConfirmed результаты;
- сценарии 15/14/13/void работают;
- админ-панель показывает provider health и споры;
- пользователь видит понятный статус результата.

## 32. Итоговое решение

Для этапа 7 фиксируется:

- стартовый продукт остается `Суперэкспресс 15`;
- основной фокус на футболе, но терминология остается `спортивный экспресс`;
- для денежных тиражей нужен 1 основной API и 2 резервных API;
- конкретные production-провайдеры выбираются после проверки лицензии, качества и SLA;
- API-Football можно использовать для MVP/dev-статистики и первичной интеграции, если это разрешено тарифом и договором;
- Chainlink Functions не используем как production-зависимость;
- целевой путь после MVP - Chainlink CRE;
- результат считается только после consensus, апелляции и финализации;
- manual fallback разрешен только как аварийная процедура;
- события нельзя заменять после открытия продаж;
- при 1-2 проблемных событиях возможен сокращенный тираж 14/13;
- при 3+ проблемных событиях тираж void, ставки возвращаются;
- статистика вероятностей является информационной и не влияет на расчет выигрышей;
- реальное распределение ставок OX можно показывать осторожно, чтобы не усиливать кита.

Главный принцип:

```text
быстрая выплата допустима только после надежного результата.
```
