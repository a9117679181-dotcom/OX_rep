# OX. Техническое задание. Этап 2: Жизненный цикл тиража

Дата: 2026-06-08  
Проект: Onchain Express / OX  
Документ: этап 2 технического задания  
Статус: черновик для согласования

## 1. Назначение этапа

Цель этапа 2 - описать полный жизненный цикл тиража OX: от создания тиража до закрытия выплат и архивирования результатов.

Этот этап отвечает на вопросы:

- кто создает тираж;
- как выбираются события;
- какие параметры фиксируются до открытия продаж;
- когда пользователь может покупать билеты;
- когда продажи закрываются;
- как поступают результаты спортивных событий;
- когда запускается расчет победителей;
- как работает окно апелляции;
- когда начинаются автоматические выплаты;
- как тираж закрывается и попадает в архив.

## 2. Базовая логика тиража

Тираж - это отдельный раунд OX, в котором пользователи покупают билеты на набор спортивных событий.

Для MVP используется продукт:

`Суперэкспресс 15`

Базовые параметры:

- 15 спортивных событий;
- стартовый спорт - футбол;
- 3 исхода на событие;
- пользователь покупает билет до закрытия продаж;
- после завершения событий система получает результаты;
- система рассчитывает победителей;
- выплаты проходят автоматически для пользователя через безопасную техническую модель.

В будущем должен быть добавлен:

`Мегаэкспресс 16`

Поэтому жизненный цикл тиража должен поддерживать разные типы тиражей через параметр:

`eventsCount (количество событий)`

## 3. Основные состояния тиража

Тираж должен проходить через фиксированные состояния.

| Статус | Простое описание | Что происходит |
|---|---|---|
| Draft (черновик) | Тираж готовится | Администратор создает тираж, добавляет события, проверяет параметры. |
| Scheduled (запланирован) | Тираж готов к запуску | Параметры проверены, но продажи еще не открыты. |
| Open (открыт) | Можно покупать билеты | Пользователи покупают билеты. Ключевые параметры уже нельзя менять. |
| Locked (закрыт для покупок) | Продажи завершены | Новые билеты купить нельзя. Система ждет окончания событий. |
| Awaiting Results (ожидание результатов) | Ждем результаты | Оракул или спортивный API собирает результаты событий. |
| Results Under Review (результаты на проверке) | Результаты опубликованы, открыто окно апелляции | Пользователи и администратор видят результаты. Можно подать апелляцию в течение установленного времени. |
| Settlement Pending (ожидание расчета) | Готовим расчет | Система считает победителей и суммы выплат. |
| Settlement Published (расчет опубликован) | Список выплат опубликован | Публикуется Merkle root (корень дерева выплат). |
| Payout Timelock (таймер перед выплатами) | Идет финальная задержка перед выплатами | Система ждет установленное время перед запуском автоматических выплат. |
| Payouts Active (выплаты активны) | Выплаты доступны | Relayer (служба автоматической отправки транзакций) начинает sponsored claims (получение выплат за счет платформы). |
| Completed (завершен) | Тираж закрыт | Выплаты доступны, результаты и история сохраняются. |
| Paused (пауза) | Временная остановка | Используется при критической ошибке или риске. |
| Cancelled (отменен) | Тираж отменен | Используется только по заранее описанным правилам. |

## 4. Создание тиража

### 4.1. Кто создает тираж в MVP

В MVP тираж создает администратор через admin panel (админ-панель).

Причина:

- на старте нужно вручную контролировать качество событий;
- нужно проверять время матчей;
- нужно учитывать риск переносов и отмен;
- нужно быстрее запустить MVP без сложной автоматизации;
- ручное создание проще проверить и отладить.

### 4.2. Будущая автоматизация

После MVP возможно добавить Cron (автоматический планировщик), который будет:

- предлагать список событий;
- проверять расписание;
- исключать подозрительные или неудобные события;
- готовить черновик тиража;
- передавать черновик администратору на подтверждение.

На следующем уровне автономии система сможет создавать тиражи по заранее заданным правилам, но только после тестов, аудита и стабильной работы.

## 5. Формирование списка событий

### 5.1. MVP

Для `Суперэкспресса 15` администратор выбирает 15 спортивных событий.

На старте основной тип событий:

- футбольные матчи.

Будущие дисциплины возможны, если они подходят под формат 3 исходов или могут быть формализованы в правилах OX.

Пример будущего расширения:

- хоккей можно рассматривать как отдельную дисциплину только после анализа правил, исходов и формата события.

### 5.2. Требования к событию

Каждое событие должно иметь:

- уникальный идентификатор;
- название турнира;
- название участников;
- время начала;
- ожидаемое время окончания;
- источник данных;
- доступные исходы;
- статус события;
- правила обработки переноса или отмены.

### 5.3. Запреты

Нельзя добавлять событие, если:

- неизвестно точное время начала;
- источник данных ненадежен;
- событие уже началось;
- событие имеет высокий риск отмены;
- правила исхода нельзя объяснить пользователю;
- событие запрещено для выбранной страны или рекламного рынка.

## 6. Параметры тиража

До открытия продаж должны быть зафиксированы:

- тип продукта: `Суперэкспресс 15` или другой тип;
- `eventsCount (количество событий)`;
- список событий;
- доступные исходы;
- валюта покупки;
- сеть;
- минимальная ставка;
- максимальные лимиты, если применяются;
- время открытия продаж;
- время закрытия продаж;
- правила отмены события;
- правила переноса события;
- правила апелляции;
- окно апелляции;
- правила выплат;
- адреса контрактов;
- правила emergency pause (экстренной паузы).

## 7. Неизменяемость параметров

Ключевое правило:

`После открытия продаж ключевые параметры тиража менять нельзя.`

Нельзя менять после статуса Open (открыт):

- список событий;
- количество событий;
- доступные исходы;
- заменять одно событие другим событием;
- правила выигрыша;
- правила джекпота;
- распределение призового фонда;
- minBet (минимальную ставку);
- время закрытия продаж, если изменение ухудшает положение пользователя;
- валюту;
- сеть;
- правила апелляции;
- правила выплат.

Можно менять только технические и безопасные параметры, если они не ухудшают положение пользователя:

- отображаемое описание;
- исправление явной опечатки в названии;
- служебный статус мониторинга;
- внутренний комментарий администратора.

Все такие действия должны попадать в audit log (журнал действий).

Если после открытия продаж событие отменено или перенесено, администратор не заменяет его новым событием. Применяются правила обработки отмены и переноса событий.

## 8. Открытие продаж

Продажи открываются только если:

- выбран полный список событий;
- все события прошли проверку;
- параметры тиража зафиксированы;
- смарт-контракт готов принимать билеты;
- админ-панель показывает корректный статус;
- мониторинг работает;
- relayer (служба автоматических транзакций) активен;
- есть достаточный gas reserve (резерв на оплату транзакций);
- правила тиража опубликованы пользователю.

После открытия продаж пользователь может:

- подключить кошелек;
- выбрать исходы;
- купить билет;
- купить системную ставку, если она разрешена в MVP;
- видеть стоимость до подтверждения;
- видеть купленный билет в истории.

## 9. Закрытие продаж

Продажи закрываются:

- по заранее заданному времени;
- до начала первого события;
- вручную администратором только если это предусмотрено правилами и не нарушает права пользователей;
- автоматически системой, если наступило время закрытия.

После закрытия продаж:

- новые билеты купить нельзя;
- параметры тиража остаются неизменными;
- система ждет завершения событий;
- купленные билеты остаются в истории пользователя;
- призовой фонд фиксируется для дальнейшего расчета.

## 10. Получение результатов

Результаты спортивных событий поступают через oracle flow (процесс передачи данных через оракул).

Для MVP:

- целевой путь - Chainlink CRE (Chainlink Runtime Environment, среда выполнения Chainlink для workflow/процессов);
- Chainlink Functions не используется как production-зависимость, потому что сервис уходит в sunset (вывод из эксплуатации) 1 сентября 2026 года;
- CRE поддерживает workflow (процессы) на Go или TypeScript, а также симуляцию перед production-деплоем;
- API-Football рассматривается как стартовый источник футбольной статистики и тестовых данных;
- production-источники результатов для денежных тиражей должны быть выбраны отдельно в этапе оракула;
- конкретный набор sports data providers (поставщиков спортивных данных) нельзя фиксировать без проверки лицензий, доступности, стоимости и качества данных.

Для production-результатов должна использоваться схема из трех источников:

- primary API (основной источник спортивных данных);
- backup API 1 (первый резервный источник);
- backup API 2 (второй резервный источник).

Принцип работы:

- основной источник используется первым;
- два резервных источника нужны для проверки, сравнения и аварийного переключения;
- если основной источник недоступен или дает подозрительный результат, система сверяет данные с резервными источниками;
- если источники расходятся, событие переводится в статус проверки и может потребовать manual fallback (ручного резервного сценария);
- конкретные API-провайдеры выбираются позже после проверки лицензий, качества данных, покрытия матчей, стоимости и технической доступности.

Важное правило:

`Этап 2 фиксирует жизненный цикл получения результатов, но финальную oracle-архитектуру и список провайдеров данных мы утверждаем в отдельном этапе "Оракул и результаты матчей".`

Результат события должен иметь:

- идентификатор события;
- финальный счет или финальный исход;
- время фиксации результата;
- источник результата;
- статус подтверждения;
- ссылку/доказательство источника, если доступно.

## 11. Публикация результатов

После получения результатов система переводит тираж в статус:

`Results Under Review (результаты на проверке)`

На этом этапе:

- результаты видны пользователям;
- система еще не начинает финальные выплаты;
- открывается окно проверки и апелляции;
- администратор и пользователи могут увидеть спорные события.

## 12. Апелляция и окно проверки

Базовое правило:

- 6 часов на апелляцию после публикации результатов;
- если апелляций нет или они отклонены, после этого запускается еще 6 часов таймера перед выплатами;
- после завершения второго окна выплаты становятся активными.

Цель окна апелляции:

- исправить ошибку источника данных;
- обработать спорный матч;
- проверить перенос или отмену события;
- не допустить неверных выплат.

Кто может инициировать проверку:

- пользователь через интерфейс;
- администратор;
- оператор результатов;
- автоматический мониторинг.

Решение по критическим спорным случаям должно проходить через multisig (мультиподписной кошелек), если оно влияет на результаты или выплаты.

### 12.1. Если апелляция подана

Если апелляция подана:

- она фиксируется в журнале действий;
- пользователь видит статус апелляции;
- администратор или оператор результатов проверяет спорный случай;
- если решение влияет на результат события или выплаты, оно подтверждается через multisig 2 из 3;
- решение должно быть принято в пределах окна апелляции или через явно зафиксированное продление.

### 12.2. Если результаты скорректированы

Если апелляция принята и результаты изменены:

- публикуется новая версия результатов;
- старая версия остается в истории;
- окно апелляции запускается заново;
- максимальное количество повторных окон - 2.

Если после двух повторных окон спор не закрыт, тираж переводится в Paused (пауза), а multisig должен принять одно из решений:

- финализировать результаты;
- продлить разбор как исключение;
- отменить тираж по правилам void (несостоявшийся тираж);
- применить заранее описанные правила исключения события.

### 12.3. Таймер перед выплатами

Если апелляций нет или они отклонены:

- тираж переходит к расчету;
- публикуется расчет;
- включается статус `Payout Timelock (таймер перед выплатами)`;
- после 6 часов автоматические выплаты становятся активными.

Цель таймера:

- дать системе время на проверку расчета;
- дать мониторингу время обнаружить критическую ошибку;
- не запускать выплаты мгновенно при спорном результате;
- сохранить честность и прозрачность процесса.

## 13. Manual fallback

`Manual fallback (ручной резервный сценарий)` используется, если автоматический oracle flow не сработал или дал спорный результат.

Manual fallback разрешен только если:

- событие завершено;
- автоматический источник не дал результат;
- источник дал явно неверный результат;
- есть подтверждение из надежных внешних источников;
- действие фиксируется в журнале;
- критическое действие подтверждается multisig 2 из 3.

Manual fallback не должен быть обычным режимом работы. Это аварийная процедура.

Пользователь должен видеть, что тираж или событие прошло через ручную проверку.

## 14. Расчет победителей

После завершения окна апелляции система переходит к расчету.

Расчет включает:

- проверку всех купленных билетов;
- сравнение прогнозов с финальными исходами;
- определение уровней выигрыша;
- расчет сумм выплат;
- подготовку списка победителей;
- формирование Merkle tree (дерева выплат);
- публикацию Merkle root (корня дерева выплат).

Важно:

На этапе 2 мы фиксируем только порядок расчета. Формулы призового фонда, уровни выплат и джекпот будут детализированы в этапе 4.

## 15. Публикация расчета

После расчета тираж получает статус:

`Settlement Published (расчет опубликован)`

Публикуется:

- список уровней выигрыша;
- количество победителей по уровням;
- общая сумма выплат;
- Merkle root;
- статус готовности выплат.

Пользователь должен видеть:

- выиграл билет или нет;
- сколько исходов угадано;
- сумма выплаты, если билет выиграл;
- статус выплаты.

## 16. Выплаты

После публикации расчета тираж сначала переходит в статус:

`Payout Timelock (таймер перед выплатами)`

После завершения таймера тираж переходит в статус:

`Payouts Active (выплаты активны)`

Базовая модель:

- relayer (служба автоматической отправки транзакций) выполняет sponsored claims (получение выплат за счет платформы);
- пользовательский опыт выглядит как автоматическая выплата;
- если автоматическая выплата не прошла, пользователь может использовать manual claim fallback (ручное получение выплаты).

На этапе 2 фиксируется порядок. Детальная архитектура выплат будет описана в этапе 6.

## 17. Закрытие тиража

Тираж считается завершенным, когда:

- результаты зафиксированы;
- расчет опубликован;
- выплаты активированы;
- информация сохранена в истории;
- спорные ситуации закрыты;
- журнал действий сохранен.

Статус:

`Completed (завершен)`

После завершения пользователь может:

- открыть архив тиража;
- увидеть события;
- увидеть результаты;
- увидеть общие показатели;
- увидеть свои билеты;
- увидеть статус выплат.

## 18. Отмена тиража

Отмена тиража - крайняя мера.

Причины возможной отмены:

- критическая ошибка до открытия продаж;
- большое количество отмененных событий;
- сбой смарт-контракта;
- юридический или санкционный риск;
- критическая ошибка источника данных;
- emergency pause (экстренная пауза), после которой продолжение невозможно.

Если тираж отменяется после покупки билетов, должны быть описаны правила возврата средств.

### 18.1. Базовые правила отмены и переноса событий

Для `Суперэкспресса 15` предварительно фиксируются такие lifecycle-правила:

| Ситуация | Базовое правило |
|---|---|
| Отменено 3 или более событий из 15 | Тираж признается несостоявшимся: void (аннулирование), ставки возвращаются пользователям. |
| Отменено 1-2 события из 15 | Тираж может продолжиться с уменьшенным количеством событий: 14 или 13. |
| Событие перенесено в пределах 48 часов | Событие может остаться в тираже, если это не нарушает сроки расчета и правила пользователя. |
| Событие перенесено более чем на 48 часов | Событие исключается, дальше применяются правила для уменьшенного количества событий. |
| Событие началось раньше времени закрытия продаж | Тираж должен быть поставлен на паузу для проверки. |

### 18.2. Правила для уменьшенного количества событий

Если тираж продолжается с 14 или 13 событиями:

- пользовательские билеты остаются действительными по оставшимся событиям;
- исключенное событие считается void (исключенным из расчета);
- каскад выплат должен быть сдвинут под новое количество событий;
- джекпот в таком тираже предварительно не разыгрывается, если иное не будет согласовано в этапе призового фонда;
- точные формулы и уровни выплат фиксируются в этапе 4 "Призовой фонд и джекпот".

Почему не фиксируем всю математику здесь:

- уменьшение количества событий меняет вероятности;
- сдвиг каскада влияет на экономику;
- нужно отдельно проверить, не создает ли это уязвимость или несправедливость для игроков.

### 18.3. Запрет замены события

После открытия продаж событие нельзя заменить другим событием.

Причина:

- пользователь покупал билет на конкретный набор событий;
- замена события после покупки разрушает доверие;
- замена может изменить вероятность выигрыша;
- это может выглядеть как ручное вмешательство в правила.

Если событие отменено или перенесено, применяются правила void/исключения, а не замена.

Эти правила будут дополнительно детализированы в этапах призового фонда, смарт-контрактов, оракула и выплат.

## 19. Пауза тиража

Статус:

`Paused (пауза)`

Пауза используется при:

- подозрении на ошибку;
- сбое оракула;
- атаке;
- проблеме relayer;
- проблеме контракта;
- юридическом сигнале;
- аномальной активности.

Пауза не должна давать администратору право менять правила открытого тиража.

Пауза нужна только для защиты пользователей и средств.

## 20. Мониторинг тиража

Система должна отслеживать:

- время открытия продаж;
- время закрытия продаж;
- статус каждого события;
- задержку результатов;
- ошибки oracle flow;
- состояние relayer;
- баланс gas;
- аномальную покупку билетов;
- попытки full coverage (покупки всех комбинаций);
- состояние settlement engine;
- очередь выплат;
- ошибки claim (получения выплаты).

Критические события должны создавать alert (уведомление/сигнал тревоги) для команды.

### 20.1. Аномальная активность

Система должна отдельно отслеживать:

- попытку full coverage (покупки всех комбинаций);
- резкий всплеск ставок с одного адреса;
- резкий всплеск ставок с группы связанных адресов;
- необычно крупную системную ставку;
- попытки покупки в последние секунды перед закрытием;
- превышение настроенных лимитов;
- подозрительные повторяющиеся транзакции.

При обнаружении аномалии система должна:

- отправить alert (уведомление/сигнал тревоги) команде;
- записать событие в журнал;
- показать статус в админ-панели;
- при превышении критического порога временно перевести тираж в `Paused (пауза)`.

Пороговые значения на старте задаются в админ-панели и уточняются после анализа реального поведения пользователей.

Важно:

Пауза из-за мониторинга не дает права менять правила открытого тиража. Она нужна только для проверки и защиты пользователей.

## 21. Журнал действий

Все важные действия должны сохраняться в audit log (журнале действий):

- создание тиража;
- изменение черновика;
- открытие продаж;
- закрытие продаж;
- получение результатов;
- ручной ввод результата;
- запуск апелляции;
- решение спорного события;
- публикация расчета;
- запуск выплат;
- пауза;
- отмена;
- действия multisig.

Предварительное правило хранения:

- логи хранятся 5 лет.

Финальный срок хранения должен быть проверен юристами по странам запуска.

### 21.1. Где хранятся логи

Журнал действий должен храниться в двух слоях:

| Слой | Что хранится |
|---|---|
| Blockchain events (события в блокчейне) | События смарт-контрактов: открытие тиража, покупка билетов, публикация root, claims, пауза, важные on-chain действия. |
| Database logs (логи в базе данных) | Действия администратора, апелляции, manual fallback, мониторинг, алерты, технические ошибки, комментарии операторов. |

Требования:

- действия администратора не должны удаляться бесследно;
- изменения должны иметь автора, время и причину;
- спорные решения должны иметь ссылку на источник или комментарий;
- действия multisig должны быть связаны с конкретным событием в журнале;
- экспорт логов должен быть доступен для аудита.

## 22. Путь к автономии после MVP

В MVP жизненный цикл тиража управляемый.

После MVP OX должен двигаться к большей автономии:

1. Администратор вручную создает тиражи.
2. Cron-job (автоматическая задача по расписанию) собирает список потенциальных событий из API и готовит черновик.
3. Администратор проверяет и подтверждает черновик.
4. Система автоматически создает черновики по заранее заданным правилам.
5. Администратор участвует только в проверке исключений.
6. Multisig и timelock контролируют критические изменения.
7. Большая часть жизненного цикла работает автономно.
8. Ручное вмешательство остается только для аварийных ситуаций.

Это соответствует стратегическому принципу:

`OX развивается от контролируемого MVP к максимально автономной и децентрализованной системе.`
