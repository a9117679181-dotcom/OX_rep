# OX - Этап 15. Дорожная карта и задачи разработки

Дата: 10.06.2026  
Версия: v2  
Статус: исправленная версия с русским основным языком  
Проект: OX, спортивный экспресс  
Стартовый продукт: Суперэкспресс 15  
Будущий продукт: Мегаэкспресс 16  
Девиз: Честно. Прозрачно. Мгновенно.

---

## 1. Назначение документа

Этот документ фиксирует дорожную карту разработки OX: порядок работ, зависимости, примерные сроки, состав MVP, направления после MVP и критерии готовности.

Языковое правило документа: основной текст пишется на русском языке. Английские термины используются только как техническая подсказка в скобках или как неизбежные названия технологий, контрактов и стандартов.

Этап 15 собирает решения этапов 1-14 в практический план разработки.

Главная формула этапа:

```text
Сначала запускаем безопасный Суперэкспресс 15 с проверяемыми выплатами,
потом расширяем автоматизацию, партнерку, аналитику, продукты и токен-направление.
```

Документ нужен для пяти задач:

- определить, что разрабатываем сначала;
- показать зависимости между блоками;
- разделить MVP, первый рабочий запуск и развитие после MVP;
- дать список задач разработки;
- определить критерии приемки по этапам.

---

## 2. Базовое решение

Принимаем:

```text
MVP OX - это Суперэкспресс 15 на Base с USDC, серверным агрегатором результатов, ручным резервным сценарием, расчетом выплат через дерево Merkle, автоматическими выплатами за счет платформы, админ-панелью, журналом аудита и контрольным порогом безопасности.
```

В MVP не входит:

- Мегаэкспресс 16;
- токен OX;
- механика DEX/listing/token (децентрализованная биржа, листинг, токен);
- полностью автономный oracle через Chainlink CRE;
- массовый RevShare;
- внешний партнерский кабинет как обязательный блок;
- публичная программа поиска уязвимостей до исправления замечаний аудита;
- полная децентрализация управления.

## 4. Общая последовательность разработки

Рекомендуемый порядок:

1. Фиксация продукта и архитектуры.
2. Основные смарт-контракты.
3. Серверная часть и база данных.
4. Индексатор событий блокчейна.
5. Оракул и серверный агрегатор результатов.
6. Расчет тиража и конвейер выплат.
7. MVP админ-панели.
8. MVP пользовательского интерфейса.
9. Мониторинг, журнал аудита и резервное копирование.
10. Комплаенс, геоограничения и KYC/AML-интеграции.
11. Тесты безопасности и прогон в тестовой сети.
12. Внешний аудит и исправление замечаний.
13. Ограниченный запуск в mainnet.
14. Минимальная партнерская атрибуция.
15. Расширение после MVP.

Почему именно так:

- без смарт-контрактов нельзя корректно строить серверные события;
- без серверной части и индексатора нельзя проверить выплаты;
- без оракула и расчета тиража нельзя пройти полный цикл;
- без админ-панели нельзя безопасно управлять MVP;
- без контрольного порога безопасности нельзя выходить в mainnet;
- партнерку и направление DEX/token нельзя ставить раньше основного продукта, чтобы не перегрузить запуск.

## 6. Этапы разработки

### Фаза 0. Подготовка

Цель: зафиксировать базовую архитектуру до разработки.

Задачи:

- подтвердить финальный состав MVP;
- выбрать технологический стек;
- выбрать сеть для разработки и тестовую сеть;
- подтвердить Base + native USDC как стартовое решение;
- определить репозитории;
- определить CI/CD;
- определить роли команды;
- подготовить архитектурную схему;
- подготовить черновик модели данных;
- подготовить чеклист безопасности.

Оценка:

```text
1-2 недели
```

Критерии приемки:

- состав работ подписан;
- архитектурная схема готова;
- список задач разбит на рабочие задачи;
- окружение разработки работает;
- нет критических открытых вопросов, блокирующих старт.

### Фаза 1. Смарт-контракты MVP

Цель: реализовать блокчейн-основу продукта.

Задачи:

- DrawManager;
- TicketVault;
- TreasuryVault;
- ResultRegistry;
- PayoutVault;
- логика возвратов;
- роли доступа;
- пауза;
- события контрактов;
- проверка доказательства Merkle;
- карта уже полученных выплат;
- неизменяемые MVP-контракты без обновляемой логики;
- модульные тесты;
- инвариантные тесты;
- fuzz-тесты;
- статический анализ.

Оценка:

```text
3-5 недель
```

Критерии приемки:

- контракты проходят модульные тесты;
- инварианты проходят;
- fuzz-тесты проходят;
- вывод пользовательских средств администратором невозможен;
- повторное получение выплаты невозможно;
- параметры тиража нельзя менять после статуса Open;
- события достаточны для индексатора.

### Фаза 2. Серверная часть

Цель: реализовать серверную основу.

Задачи:

- схема базы данных;
- публичный API;
- административный API;
- подписанное разрешение на покупку;
- риск-движок;
- сервис жизненного цикла тиража;
- сервис билетов;
- сервис расчета тиража;
- построитель дерева Merkle;
- API доказательств;
- API статусов;
- сервис журнала аудита;
- идемпотентность;
- задания, устойчивые к повторному запуску.

Оценка:

```text
4-6 недель
```

Критерии приемки:

- API покрыты интеграционными тестами;
- подписанное разрешение на покупку работает;
- расчет тиража не запускается до финального результата;
- корень выплат не строится без отчета о расчете;
- proof API не раскрывает чужие данные;
- журнал аудита создается для критических действий.

### Фаза 3. Индексатор

Цель: связать события блокчейна с состоянием серверной части.

Задачи:

- обработчики событий;
- контрольная точка обработки;
- повторное воспроизведение событий;
- обработка реорганизаций блоков;
- сверка базы данных и блокчейна;
- защита от дублей;
- мониторинг задержки индексатора;
- восстановление после сбоя.

Оценка:

```text
2-4 недели
```

Критерии приемки:

- TicketPurchased индексируется;
- PayoutClaimed индексируется;
- повторное воспроизведение не создает дубли;
- реорганизация на несколько блоков обрабатывается;
- падение во время записи не ломает состояние;
- задержка индексатора вызывает предупреждение.

### Фаза 4. Оракул и агрегатор результатов

Цель: реализовать получение и проверку спортивных результатов.

Задачи:

- адаптер основного поставщика данных;
- адаптеры резервных поставщиков;
- сопоставление спортивных событий;
- регулярное получение результатов;
- логика консенсуса и расхождений;
- манифест результатов;
- хэш исходных данных;
- окно апелляции;
- ручной резервный сценарий;
- финализация результата;
- мониторинг состояния поставщиков.

Оценка:

```text
4-6 недель
```

Критерии приемки:

- 3 источника данных поддержаны архитектурно;
- расхождение источников блокирует расчет тиража;
- ручной резервный сценарий требует код причины, доказательства и запись в журнале аудита;
- решения, влияющие на выплаты, требуют multisig 2 из 3;
- манифест результатов версионируется.

### Фаза 5. Конвейер выплат и релейер

Цель: реализовать автоматические выплаты за счет платформы.

Задачи:

- очередь выплат;
- сервис исполнителя выплат;
- функция claimFor;
- мониторинг резерва газа;
- обработка уже полученных выплат;
- ручной сценарий получения выплаты пользователем;
- сценарий возвратов;
- синхронизация статусов выплат;
- резервная копия доказательств;
- резервная копия манифеста выплат.

Оценка:

```text
3-5 недель
```

Критерии приемки:

- исполнитель не делает двойную выплату;
- пауза при нехватке газа работает;
- ручное получение выплаты пользователем работает;
- возврат при отмененном тираже работает;
- доказательство и манифест сохраняются;
- статус выплаты обновляется после подтверждения в блокчейне.

### Фаза 6. MVP админ-панели

Задачи:

- MFA;

Критерии приемки:

## 7. Параллельность работ

Можно делать параллельно:

- смарт-контракты и модель данных серверной части;
- прототип интерфейса и контракты API;
- каркас админ-панели и административный API;
- адаптеры оракула и сопоставление событий;
- настройку мониторинга и серверные сервисы;
- планирование тестов безопасности и разработку контрактов.

Нельзя делать раньше зависимостей:

- исполнитель выплат до формата доказательства Merkle;
- расчет тиража до финального манифеста результатов;
- развертывание в mainnet до исправления замечаний аудита;
- партнерские выплаты до финансовой модели;
- DEX/token до отдельного решения после MVP;
- миграцию на Chainlink CRE до стабильной версии серверного агрегатора.

- архитектурная поддержка 3 источников спортивных данных;

- окно апелляции;
- расчет выплат через дерево Merkle;

- админ-панель;
- журнал аудита;
- мониторинг;
- резервное копирование и восстановление;

В MVP не входит:

- мобильное приложение.

## 9. Дорожная карта после MVP

Направления после MVP:

| Направление | Когда делать | Комментарий |
|---|---|---|
| Мегаэкспресс 16 | после стабильного Суперэкспресс 15 | увеличивает сложность и джекпот |
| Chainlink CRE | после аудита и стабильного оракула | путь к большей автономности |
| Автоматический выбор событий по расписанию | после контроля ручного управления событиями | не блокирует MVP |
| Расширенная аналитика событий | после UX проверки | не перегружать первый экран |
| Партнерский кабинет | после базовой CPA-атрибуции | не блокирует основной запуск |
| RevShare | после финансовой модели | только проверенным партнерам |
| Дополнительные страны | после юридической проверки | страна за страной |
| Дополнительные языки | после выбора стран | конвейер локализации |
| Режим высоких ставок | после симулятора, KYC и риск-проверки | не для MVP |
| Публичная программа поиска уязвимостей | после исправления замечаний внешнего аудита | сначала закрытая проверка |
| DEX/listing/token | отдельный стратегический этап | без разработки токена в MVP |

## 10. Направление DEX/listing/token

Фиксируем:

```text
Токен OX не входит в MVP.
DEX/listing/token не должны усложнять первый запуск.
```

Что можно делать на этапе MVP:

- не блокировать будущую интеграцию токена архитектурно;
- не смешивать токен с призовым фондом;
- не обещать токен пользователям;
- не строить выплаты или партнерку на токене;
- вести отдельный стратегический список задач.

Что нельзя делать в MVP:

- выпускать токен OX;
- обещать листинг;
- использовать токен как платежную валюту;
- использовать токен для выплат;
- использовать токен как гарантию выигрыша;
- завязывать смарт-контракты MVP на будущий токен.

Решение:

```text
DEX/listing/token - направление после MVP, допустимое только после юридической, финансовой проверки и проверки безопасности.
```

## 15. Чеклист готовности к запуску

Перед mainnet должны быть готовы:

- контракты прошли аудит;

- прогон в тестовой сети завершен;

- мультиподпись настроена;
- ключи подписантов разделены;
- серверная часть готова к рабочему запуску;
- повторное воспроизведение индексатора протестировано;
- расхождение источников оракула протестировано;
- исполнитель выплат протестирован;
- учебное восстановление из резервной копии завершено;
- предупреждения мониторинга настроены;
- регламенты реагирования на инциденты готовы;
- страны запуска одобрены юридически и комплаенс-командой;
- пользовательские условия и предупреждения о рисках готовы;
- операторы админ-панели обучены;
- коммуникация запуска подготовлена.

## 16. Что считается готовым

Компонент считается готовым, если:

- функциональность реализована;
- тесты проходят;
- негативные сценарии проверены;
- логи и статусы работают;
- мониторинг подключен;
- риски безопасности учтены;
- документация обновлена;
- критерии приемки выполнены;
- ответственный подтвердил готовность.

MVP считается готовым к тестовой сети, если:

- основные контракты работают;
- серверная часть, индексатор, оракул и конвейер выплат связаны;
- админ-панель и пользовательский интерфейс проходят полный сценарий;
- тестовые данные готовы;
- есть мониторинг;
- есть журнал аудита;
- есть резервное копирование;
- есть план тестирования.

MVP считается готовым к mainnet, если:

- прогон в тестовой сети пройден;
- внешний аудит пройден;
- проверка исправлений пройдена;
- контрольный порог безопасности подписан;
- юридический и комплаенс-состав запуска подтвержден;
- реагирование на инциденты готово.

## 17. Риски дорожной карты

| Риск | Последствие | Митигирующее действие |
|---|---|---|
| Затянулась юридическая проверка | нельзя запускать страны | начинать legal-процесс параллельно с разработкой |
| Поставщики данных оракула нестабильны | расчет тиража задерживается | 3 источника и ручной резервный сценарий |
| Критические замечания аудита | запуск переносится | ранние инвариантные и fuzz-тесты |
| Расхождение серверной части и корня выплат | неверные выплаты | сверка и прогон в тестовой сети |
| Потеря доказательств или манифеста | нельзя доказать выплату | холодная резервная копия |
| Сбой газа или исполнителя выплат | задержка выплат | мониторинг газа и ручное получение выплаты |
| UI перегружен | низкая конверсия | MVP UI без лишней аналитики |
| Партнерка перегружает запуск | расползание состава работ | партнерка после основного запуска |
| Ажиотаж вокруг токена отвлекает | юридический и продуктовый риск | токен только после MVP |
